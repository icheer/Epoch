# Epoch UI/UX 全面审查报告

**版本**: 1.0 | **日期**: 2026-05-15 | **工具**: Impeccable (LLM 设计审查 + 代码级扫描)

---

## 概述

Epoch 是一款将 LLM 对话转化为交互式组件 UI 的工具型产品（product register）。架构设计极具新意 — LLM 生成 JSON 组件树，客户端实时渲染为可交互界面。然而，当前 UI 的实现质量远未匹配这个创意的野心。视觉语言依赖 AI 设计刻板印象，无障碍性几乎为零，用户控制权严重不足。

## AI Slop 判定

**结论：是。** 不需要第二眼。

当前界面有三个一级 AI 设计刻板印象：

1. **粉→黄渐变** (`from-pink-500 to-yellow-500`) 出现在发送按钮、助手头像、ButtonRenderer 主变体，三处重复。这个配色组合在 2024-2025 年的 AI/SaaS 产品中极为泛滥，是最典型的"AI 生成设计"识别符。
2. **英雄指标模板**：MetricRenderer 是教科书式的被禁反模式 — 大号数字 + 小标签 + 趋势图标 + 变体颜色背景，完全符合"SaaS 大盘仪表板"刻板印象。
3. **相同卡片网格**：StatsRenderer 渲染 4 个结构完全相同的白色卡片（图标 + 值 + 标签 + 变化量），无任何视觉差异化。

次级告示：
- backdrop-blur-sm 应用在所有按钮变体上，即使是实心灰色背景，模糊效果无任何意义（纯 GPU 浪费）
- 整体色彩策略是无品牌个性的默认 SaaS 灰白，靠单个渐变强调色撑场，缺乏设计意图

---

## 亮点（值得保留和发扬）

1. **实时流式渲染**：SSE + 渐进渲染的技术实现印象深刻。响应感流畅，组件树在生成过程中逐步出现的体验是产品核心价值的直观体现。保留并进一步放大这个特性的视觉呈现。

2. **Geist 字体选择**：Geist Sans 和 Geist Mono 组合对开发者/技术产品来说是成熟的选择，有质感但不张扬，留有充分的设计空间。

3. **基础响应式骨架**：`max-w-3xl` 容器、移动端 padding 适配、Hero/Card 的断点设计基本合理，基础布局逻辑正确。

## 优先级问题

### P1 — 重大问题（正式发布前必须修复）

#### [P1] 无障碍性几乎缺席

**问题**: 发送按钮仅包含图标 `<ArrowUp size={13}/>` 无任何文字说明；Textarea 无 `aria-label` 或关联 `<label>`；消息列表无 `role="log"` 和 `aria-live="polite"`；整个页面无 `<main>`、`<header>` 等语义地标；流媒体期间无 `aria-busy` 状态。

**为什么重要**: 屏幕阅读器用户完全无法使用此应用。发送按钮对辅助技术而言是一个"未命名按钮"。消息更新对屏幕阅读器是无声的。

**修复方案**:
```tsx
// page.tsx — 发送按钮
<button aria-label="发送消息" ...>

// page.tsx — Textarea
<Textarea aria-label="输入消息" ...>

// page.tsx — 消息容器
<div role="log" aria-live="polite" aria-label="对话历史" ...>

// page.tsx — 根 div 改为语义 HTML
<main className="bg-white min-h-screen w-screen">
```

**建议命令**: `/impeccable audit`（技术修复后重新扫描）

---

#### [P1] 用户操作消息暴露内部实现格式

**位置**: `src/app/page.tsx:133-145`

**问题**: 用户点击 LLM 渲染的按钮后，会话历史中出现如下消息（作为用户气泡显示）：
```
User clicked button: "继续"

Form data:
- name: 张三
- age: 25
```

**为什么重要**: 这不是自然语言对话界面的预期行为。用户会困惑于"我没有说过这句话"，破坏对话的连贯性和信任感。暴露 `Form data:` 这类工程格式更是明显的实现泄露。

**修复方案**: 将按钮操作消息转换为自然语言，或者完全隐藏该消息（仅作为 API payload 发送，不在 UI 展示）。如果需要显示，将其视觉上区别于普通用户消息（如用斜体、不同颜色、或添加"操作"标识）。

---

#### [P1] 无法中断正在生成的响应

**位置**: `src/app/page.tsx:28-118`

**问题**: `isStreaming` 期间整个界面锁定，只有等待完成，没有停止按钮，没有任何中断机制。长响应（尤其含图片搜索的卡片）可能需要十几秒。

**为什么重要**: 工具型产品的核心体验是用户控制感。无法停止的生成让用户产生焦虑，尤其在 LLM 输出方向明显错误时。这是聊天界面的基础功能。

**修复方案**: 在 `streamResponse` 中使用 `AbortController`，流媒体时显示"停止"按钮替换或并排发送按钮。

---

### P2 — 次要问题（建议尽快修复）

#### [P2] 页面元数据未更新

**位置**: `src/app/layout.tsx:15-18`

**问题**:

```ts
export const metadata: Metadata = {

  description: "Generated by create next app",  // ← 模板占位符

```

**修复**:
```ts
export const metadata: Metadata = {
  title: "Epoch",
  description: "Visual interface for LLMs — turn AI conversations into interactive component-driven experiences.",

```

#### [P2] 首条消息与输入框语言不一致

**位置**: `src/app/page.tsx:18-22` vs `page.tsx:233`

**问题**: 初始 greeting 是 `"你好，今天想搜索点什么？"`（中文），输入框占位符是 `"Ask anything..."`（英文）。两处语言不一致，发出混乱信号。

**更深层的问题**: 初始消息本身写得很弱。"今天想搜索点什么"把 Epoch 定位成搜索工具，未提及其核心价值（LLM 生成交互式 UI）。

**修复方向**: 统一语言（选一种）；重写初始消息以体现产品价值，并提供 2-3 个示例提示帮助用户发现能力边界。

---

#### [P2] 深色模式有实现无入口，且实现不完整

**位置**: `src/app/globals.css:81-113`（.dark 定义）vs `src/app/page.tsx:159`（`bg-white` 硬编码）

**问题**: 深色模式 CSS 已定义，但：
- 没有切换 UI（没有主题切换按钮）
- `page.tsx` 的根 div 是 `bg-white`（硬编码，深色模式无效）
- 固定底部输入栏的渐变遮罩是 `from-white`（硬编码，深色模式会显示白色条带）
- 多个组件（MetricRenderer 的 variant 颜色、ButtonRenderer 主渐变）完全绕过 token 系统

**修复方案**:
1. 将 `bg-white` 改为 `bg-background`
2. 将 `from-white` 改为 `from-background`
3. 添加主题切换按钮（可以很小，放在角落）
4. MetricRenderer 的 variant 颜色改用语义 token

---

#### [P2] MetricRenderer 和 StatsRenderer 是被明确禁止的反模式

**位置**: `src/components/llm-components/MetricRenderer.tsx`、`StatsRenderer.tsx`

**问题**:
- **MetricRenderer**：大号数字 + 小标签 + 趋势图标 = 完全符合"英雄指标模板"（Hero Metric Template）反模式定义
- **StatsRenderer**：4 列相同结构白色卡片（icon + value + label + change）= 完全符合"相同卡片网格"（Identical Card Grid）反模式定义

这两个模式在任何 SaaS 仪表板中出现的概率接近 100%，是 AI 生成设计最典型的识别符。

**修复方向**: 这些组件不应该消失（功能有价值），而是需要重新设计视觉表现形式。例如：
- MetricRenderer：用更克制的内联数字展示替代卡片容器；去掉边框和背景变体
- StatsRenderer：用水平分割线、不同尺寸的数字、或纯排版方案替代白色卡片网格

---

#### [P2] transition-all 广泛使用 — 性能隐患

**位置**: `ButtonRenderer.tsx:43`、`CardRenderer.tsx:86`

**问题**: `transition-all` 会对 CSS 的所有可过渡属性启动过渡计算，包括那些从未改变的属性。这在移动设备或低端设备上会有明显的性能开销。

**修复**: 明确指定过渡属性：
```tsx
// 替换 transition-all duration-200
// 改为 transition-[shadow,transform,opacity] duration-200
```

---

#### [P2] 图片未经 Next.js 优化

**位置**: `CardRenderer.tsx:102`、`HeroRenderer.tsx:87`

**问题**: 使用原生 `<img>` 而非 Next.js `<Image>`，无自动压缩、格式转换（WebP/AVIF）、懒加载，也无尺寸优化。

**注意**: 这些图片来自外部搜索结果 URL，使用 `<Image>` 时需配置 `next.config` 的 `remotePatterns`，或保持 `<img>` 但至少添加 `loading="lazy"`。

---

### P3 — 打磨级问题（时间允许时改进）

#### [P3] 发送按钮的粉→黄渐变需要品牌化重设计

**位置**: `page.tsx:240`、`ButtonRenderer.tsx:18`

`from-pink-500 to-yellow-500` 是 2024-2025 年 SaaS/AI 产品中重复率最高的渐变组合。它没有与 Epoch 这个名字（"纪元"）或产品概念（时间轴、节点、交互界面）产生任何关联。

**替代方向**: Epoch 的名字天然关联星空、时间、技术感。可以考虑深蓝→青绿（宇宙感）、或单纯的高对比纯色（更有品质感）、或干脆用黑色按钮配白色图标（极简、不出错）。

---

#### [P3] 卡片图片悬停动画 700ms 太慢

**位置**: `CardRenderer.tsx:103`

`transition-transform duration-700 group-hover:scale-105` — 700ms 是感知上的慢，hover 触发到动画结束之间有明显延迟感。150-300ms 是 hover 动画的舒适区间。

---

#### [P3] 无会话持久化 — 刷新即消失

当前所有对话状态存于 React useState，刷新即清空。对于探索性工具而言，这极大降低了用户深入探索的意愿。建议至少实现 localStorage 持久化（`useEffect` 监听 messages 写入）。

---

#### [P3] 错误状态缺乏恢复路径

**位置**: `page.tsx:107-116`

```tsx
// 当前错误处理
content: "Sorry, an error occurred while processing your request."
```

问题：英文硬编码；无重试按钮；无错误类型区分（网络错误？模型超时？配置问题？）；出错后用户只能手动重新输入。

**修复**: 在错误消息后添加"重试"按钮，重发上一条消息；根据错误类型给出不同提示。

---

#### [P3] 流媒体加载状态可以更有信息量

**位置**: `page.tsx:214-217`

当前: 小 spinner + "Streaming..."。

改进方向: 区分不同阶段（"思考中..."→"生成界面..."→"搜索图片..."），或根本上去掉文字，让组件树的渐进出现本身传达进度感。

---

#### [P3] 无空状态设计

首次打开时直接显示 greeting，没有引导用户发现 Epoch 能力的机制。建议在 greeting 下方展示 3-5 个可点击的示例提示，覆盖不同组件类型（图表、卡片、表单等），让用户立即体验产品核心价值。

---

#### [P3] 助手 Avatar 尺寸不一致

**位置**: `page.tsx:169-171`

```tsx
<Avatar className="size-6 mt-4 shrink-0">   {/* 24×24px 容器 */}
  <div className="... h-8 w-8 rounded-full"> {/* 32×32px 内容 */}
```

内容尺寸 (32px) 大于容器 (24px)，渐变圆形被 Avatar 裁切，显示异常。

## Persona 红旗分析

### Alex — 技术用户，想深度使用 Epoch

**主要用户动作**: 尝试构建复杂的多步骤交互界面

**红旗**:
- 流媒体进行中完全锁定，无法提前输入下一个指令 — 强制串行工作流

## 系统性问题（非个案）

以下问题在代码库中系统性存在，修复需要全局处理而非逐个修补：

1. **颜色硬编码**: `bg-white`、`text-gray-X`、`border-gray-X` 等硬编码颜色散布于 `page.tsx` 和多个 LLM 组件中，完全绕过了 globals.css 中的 token 系统。凡需要支持深色模式，必须全局替换。

2. **无障碍性白板**: 整个应用无语义 HTML，无 ARIA 属性，无屏幕阅读器支持。这不是遗漏，而是从未被考虑过。需要系统性加入，而非逐个修补。

3. **按钮渐变反模式**: primary 变体按钮在整个应用中统一是粉→黄渐变，包括 page.tsx 的发送按钮和 ButtonRenderer。一旦决定更换，需要同步修改两处。
